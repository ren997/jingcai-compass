# 竞彩罗盘数据源选型与验证方案

## 1. 文档信息

- 文档版本：v0.1
- 最后核查日期：2026-07-22
- 当前状态：候选源调研完成，等待样本验证
- 目标：为 MVP 确定可持续的体彩官方数据源和足球亚洲让球盘数据源

> 本文记录的是候选方案和验证结论，不等同于已经取得数据授权。供应商套餐、字段和使用条款可能变化，正式接入前必须再次核查。

## 2. 当前结论

MVP 必须采用双数据源，而不是寻找一个来源包办所有数据：

1. **体彩数据源**负责确定哪些比赛属于中国体彩竞彩足球比赛池，以及官方比赛编号、开售状态、体彩让球、SP 和最终赛果。
2. **亚盘数据源**负责提供赛前亚洲让球盘口及其赔率快照，为预测特征、盘口变化和后续收益回测提供输入。

当前推荐的验证顺序：

1. 使用 **The Odds API** 免费档验证足球让球盘覆盖率。它很可能就是此前记忆中的“每月有免费额度”的数据源。
2. 使用中国体彩网前台及其网关接口验证官方比赛池和赛果同步，但在确认稳定性和使用许可前，只作为技术验证源。
3. 同时保留 **API-Football（API-Sports）** 和合规商业数据供应商作为亚盘备选。

当前不能直接宣布选型完成。The Odds API 虽然提供 `spreads`，但官方文档提示让球盘目前主要覆盖美国体育和博彩公司，必须用真实的每日竞彩池做覆盖率测试。

## 3. 术语与数据职责

### 3.1 亚洲让球盘

本文中的“亚盘”指博彩公司提供的亚洲让球市场，常见盘口包括：

- 0
- -0.25 / +0.25
- -0.5 / +0.5
- -0.75 / +0.75
- -1.0 / +1.0

每条快照至少需要记录：

- 数据供应商
- 博彩公司或盘口来源
- 主队和客队的供应商 ID
- 开赛时间
- 盘口值
- 主队赔率
- 客队赔率
- 抓取时间
- 供应商更新时间
- 是否滚球
- 原始响应哈希

### 3.2 体彩让球胜平负

体彩让球胜平负是体彩官方市场，通常使用整数让球，并且存在胜、平、负三个结算结果。它不能等同于二项结算的亚洲让球盘。

体彩源至少需要提供：

- 体彩官方比赛编号
- 体彩内部比赛 ID（如页面提供）
- 比赛日期、联赛和对阵双方
- 开赛时间
- 是否进入竞彩足球比赛池
- 销售状态
- 体彩让球数
- 胜平负和让球胜平负 SP
- 最终比分和官方赛果
- 官方数据更新时间

### 3.3 权威性规则

- 是否属于每日竞彩池，以体彩源为准。
- 官方比赛编号、体彩让球、SP 和赛果，以体彩源为准。
- 亚洲让球盘只作为外部分析数据，不得覆盖体彩字段。
- 两个来源数据不一致时保留双方原始值并产生异常记录，不静默覆盖。

## 4. 亚盘候选数据源

### 4.1 The Odds API

- 官网：<https://the-odds-api.com/>
- API 文档：<https://the-odds-api.com/liveapi/guides/v4/>
- 市场说明：<https://the-odds-api.com/sports-odds-data/betting-markets.html>
- 核查状态：2026-07-22 已核查官网和官方文档

已确认的信息：

- 免费 Starter 档为每月 500 credits。
- 免费档覆盖全部体育项目、多数博彩公司和全部市场类型。
- `GET /v4/sports/{sport}/odds` 支持 `h2h`、`spreads`、`totals` 等市场。
- `spreads` 在官方市场表中定义为 Points spread / Handicap。
- 查询成本主要按“区域数 × 市场数”计算；`GET /sports` 不消耗额度。
- 响应头提供剩余额度、已用额度和本次调用成本。

主要风险：

- 官方文档明确提示 `spreads` 和 `totals` 当前主要覆盖美国体育和博彩公司。
- 文档没有承诺其足球 `spreads` 一定是完整的亚洲让球盘，也没有承诺覆盖每天的中国竞彩比赛池。
- 免费额度适合验证，不适合高频采集盘口变化。
- 历史盘口、特定博彩公司和生产使用授权需要在接入前单独确认。

额度示例：

- 只查询 `eu` 区域和 `spreads` 市场时，一个联赛通常按 1 credit 估算。
- 每天查询 8 个联赛一次，约 240 credits/月。
- 每天查询 8 个联赛两次，约 480 credits/月，已经接近免费上限。
- 实际应只查询当天竞彩池涉及的联赛，并根据响应头动态控制额度。

当前判断：**P0 验证候选，不可直接作为生产选型。**

### 4.2 API-Football（API-Sports）

- 官网：<https://www.api-football.com/>
- 核查状态：待完成账号内接口和当前套餐验证

候选理由：

- 足球专用数据源，通常提供赛程、球队、赛果和赔率相关接口。
- 相比综合体育 API，更有可能提供足球博彩公司和具体盘口类型的细分数据。

必须实测的内容：

- 当前免费额度和赔率接口是否包含在免费档中。
- 返回的 bet/market 列表中是否存在可稳定识别的 Asian Handicap。
- 是否同时返回盘口值、主客赔率、博彩公司和供应商更新时间。
- 对竞彩池所涉及联赛的覆盖率。
- 历史盘口是否可用，以及是否允许用于模型训练和公开产品。

当前判断：**P1 备选，未验证前不在文档中记录具体免费额度。**

### 4.3 商业数据供应商

如果免费源不能达到覆盖率和稳定性要求，应采购有明确授权和 SLA 的商业体育数据服务。询价时必须明确要求：

- 足球亚洲让球盘，而不只是普通 `spread` 或欧赔。
- 开盘、即时盘和封盘前快照。
- 博彩公司标识和原始更新时间。
- 历史盘口数据及允许的使用范围。
- 中国大陆服务器访问稳定性。
- 数据缓存、展示、模型训练和衍生统计的授权边界。
- 服务可用性、延迟、补数和故障通知 SLA。

## 5. 体彩候选数据源

### 5.1 中国体彩网前台数据

- 官网：<https://www.sporttery.cn/>
- 已知网关域名：`https://webapi.sporttery.cn`
- 候选比赛池接口：`/gateway/jc/football/getMatchCalculatorV1.qry`
- 候选赛果接口：`/gateway/jc/football/getMatchResultV1.qry`
- 核查状态：接口形态待在中国大陆部署节点实测

优点：

- 最接近官方权威来源。
- 理论上可以获得体彩比赛编号、官方让球、SP、销售状态和官方赛果。

风险：

- 这是面向官方网站前台的接口，不是已确认的开放开发者 API。
- 没有稳定性、兼容性、调用额度和 SLA 承诺，字段及参数可能随时变化。
- 当前调研环境访问官网和网关时出现连接或 TLS 失败，可能存在网络区域或访问策略限制。
- 正式使用前需要确认网站条款、数据转载和商业使用许可。

接入原则：

- 用独立 Provider Adapter 隔离接口变化，不让前台字段直接进入领域模型。
- 保存每次原始响应、请求时间、状态码和内容哈希。
- 设置合理轮询频率和本地缓存，不进行分钟级高频抓取。
- 保留人工 CSV 导入作为短期故障补偿，但人工导入不能绕过审计和结算规则。

当前判断：**官方数据的首选验证方向，但在稳定性和授权确认前不能视为生产级 API。**

### 5.2 有授权的聚合供应商

如果中国体彩网前台接口不稳定或无法获得合适授权，应选择能提供体彩官方字段的商业聚合源。验收时必须确认它不是只提供通用足球赛程，而是真正提供：

- 体彩官方比赛编号
- 当日是否开售
- 体彩官方让球
- 胜平负和让球胜平负 SP
- 销售状态变化
- 官方赛果与异常状态

## 6. 双源比赛映射

体彩源和亚盘源通常不会共享比赛 ID，必须建立显式映射，禁止在查询时临时用球队名称模糊拼接。

建议新增实体：

### match_source_mappings

- id
- match_id
- provider_code
- external_match_id
- external_league_id
- external_home_team_id
- external_away_team_id
- mapping_status
- mapping_confidence
- mapping_method
- confirmed_by
- created_at
- updated_at

自动映射至少同时使用：

- 标准化后的主客队名称
- 联赛映射
- 开赛时间，允许有限时差
- 主客队方向

低置信度、开赛时间冲突或主客队疑似反转时进入人工复核队列，不得自动写入盘口。

## 7. 原始数据和快照模型

建议至少增加以下数据结构：

### raw_data_payloads

保存供应商原始响应，用于重放、排障和审计。

- provider_code
- data_type
- request_key
- requested_at
- provider_updated_at
- http_status
- payload
- payload_hash
- parse_status
- parse_error

### sporttery_pool_snapshots

保存体彩比赛池和 SP 的时间快照，不覆盖历史值。

### asian_odds_snapshots

保存每个来源、博彩公司、比赛和采集时点的盘口与赔率，不更新旧快照。

### data_sync_runs

记录每次同步的开始时间、结束时间、拉取数量、成功数量、失败数量、重试次数和额度消耗。

## 8. 两周验证计划

### 第 1～2 天：接口探测

- 注册 The Odds API 免费账号。
- 获取可用足球联赛、博彩公司和市场列表。
- 在中国大陆部署节点验证中国体彩网和网关可访问性。
- 保存脱敏后的样例响应和字段字典。

### 第 3～7 天：每日竞彩池覆盖测试

- 每天先获取体彩官方比赛池。
- 按当天涉及联赛查询 The Odds API 的 `spreads`。
- 映射体彩比赛和赔率比赛。
- 记录未覆盖、只有普通让球、盘口缺值和比赛时间不一致的样本。

### 第 8～14 天：稳定性和补数测试

- 对同一比赛至少保存开盘或首次可见快照，以及封盘前快照。
- 验证赛果同步和延期、取消、改期场次。
- 统计额度消耗、成功率、数据延迟和映射准确率。
- 用 API-Football 或一个商业样例源补测 The Odds API 未覆盖的比赛。

## 9. Go / No-Go 标准

同时满足以下条件才能进入完整 MVP 开发：

- 体彩比赛池字段完整率达到 100%。
- 体彩最终比分和赛果可自动结算率达到 100%，异常场次进入人工复核而不是丢失。
- 目标竞彩比赛的亚盘覆盖率达到 90% 以上。
- 亚盘记录包含明确的盘口、主客赔率、来源和时间戳。
- 双源自动映射准确率达到 98% 以上，其余样本可进入人工复核。
- 免费验证期间的月度预计用量不超过额度的 80%，或者已经明确付费预算。
- 数据使用条款允许缓存、模型训练和产品展示。

如果 The Odds API 的足球亚盘覆盖率低于 90%，直接切换到足球专用或商业数据源评估，不应通过扩大模糊匹配范围掩盖缺失数据。

## 10. 仍需决策的问题

- MVP 展示开盘、封盘前盘口，还是两者都展示。
- 是否指定博彩公司白名单，以及不同来源之间的优先级。
- ROI / Yield 使用体彩 SP 还是亚洲盘赔率，采用什么固定下注规则。
- 历史回测数据和上线后真实发布数据如何分开展示。
- 体彩前台数据能否获得满足长期运营的授权和稳定访问方式。
