# 竞彩罗盘 MVP 需求文档

## 1. 文档信息

- 项目名称: 竞彩罗盘
- 英文名称: JingCai Compass
- 文档版本: v0.1
- 文档日期: 2026-03-12
- 文档目标: 定义首个可上线验证版本的产品范围、边界和验收标准

## 2. 项目定位

竞彩罗盘是一个面向中国体彩竞彩足球每日比赛的数据分析与概率预测工具。

平台核心不是推荐“必中方案”，而是围绕每日官方比赛池，提供：

- 可追溯的赛前分析
- 可验证的概率输出
- 不可事后篡改的发布记录
- 自动结算的历史样本
- 长期可量化的模型表现

## 3. MVP 核心目标

MVP 只验证三件事：

1. 用户是否愿意查看针对中国体彩竞彩足球每日比赛的赛前数据分析
2. 平台是否能稳定完成发布、锁定、结算、归档这条最核心闭环
3. 在持续积累样本后，模型输出是否具备长期统计参考价值

## 4. 产品边界

### 4.1 MVP 只做

- 仅覆盖中国体彩每天发布的竞彩足球比赛
- 仅覆盖足球项目
- 仅展示赛前分析和赛后结算
- 仅提供数据分析、概率展示、历史统计与透明记录

### 4.2 MVP 不做

- 不提供投注、代下单、充值、返佣、合买、跟单
- 不提供“包赢”“必中”“稳赚”等营销表达
- 不接入支付和会员付费体系
- 不做比分、半全场、总进球数等复杂玩法
- 不做社区、晒单、评论区、聊天室
- 不做临场分钟级频繁刷新

## 5. 目标用户

### 5.1 主要用户

- 对中国体彩竞彩足球有长期关注的人群
- 更关注数据和历史验证，而不是短期情绪化推荐的人群

### 5.2 MVP 阶段用户分层

- 游客: 可查看全部公开页面和全量历史记录
- 管理员: 负责数据同步、任务监控、模型发布和异常处理

说明:

- MVP 阶段默认不做普通注册用户与会员权限体系
- 如果后续验证数据有价值，再进入账号与付费阶段

## 6. MVP 核心原则

1. 每条预测必须记录发布时间
2. 比赛开赛后预测必须锁定
3. 历史记录必须全量公开展示
4. 结算结果必须自动生成
5. 统计口径必须一致且可复核
6. 页面文案必须保持分析工具边界

## 7. MVP 功能范围

## 7.1 首页

### 目标

快速说明平台价值，并引导用户进入当天比赛和历史记录。

### 页面模块

- 平台定位说明
- 今日竞彩足球比赛数
- 今日已发布预测场次
- 历史累计预测场次
- 近7天命中率概览
- 近30天 ROI 或 Yield 概览
- 最近一次发布快照时间
- 历史记录入口

### 首页重点文案

- 仅提供竞彩足球数据分析与概率预测
- 所有历史发布记录公开可查
- 赛后自动结算，避免事后修改
- 模型结果仅供参考，不构成投注建议

## 7.2 当日比赛列表页

### 目标

按日期展示体彩当天可选竞彩足球比赛，并提供基础筛选与排序。

### 列表字段

- 比赛编号
- 联赛名称
- 开赛时间
- 主队
- 客队
- 主胜概率
- 平局概率
- 客胜概率
- 让球方向倾向
- 模型置信等级
- 发布状态
- 比赛状态

### 筛选条件

- 比赛日期
- 联赛
- 开赛时间区间
- 球队名称
- 置信等级
- 发布状态
- 比赛状态

### 排序方式

- 开赛时间
- 模型置信度
- 联赛优先级

## 7.3 比赛详情页

### 目标

展示单场竞彩足球比赛的完整赛前分析、透明信息和赛后结算结果。

### 基础信息模块

- 官方比赛编号
- 联赛
- 开赛时间
- 主队 / 客队
- 让球信息
- 比赛状态
- 最终比分
- 数据更新时间

### 模型分析模块

- 主胜概率
- 平局概率
- 客胜概率
- 让球胜平负倾向
- 预期进球数
- 模型置信等级
- 简短分析摘要

### 透明信息模块

- 预测发布时间
- 锁定时间
- 模型版本
- 预测哈希值
- 快照编号
- 是否已锁定

### 赛后结算模块

- 最终赛果
- 胜平负结算结果
- 让球胜平负结算结果
- 结算时间
- 结算说明

## 7.4 历史预测记录页

### 目标

建立平台可信度，确保历史发布全量公开、可筛选、可追溯。

### 强制要求

- 展示全量历史记录
- 不允许只展示命中场次
- 不允许删除已发布预测
- 必须保留发布时间和锁定状态

### 列表字段

- 比赛日期
- 联赛
- 对阵双方
- 官方比赛编号
- 发布时间
- 胜平负概率
- 让球方向
- 最终赛果
- 结算状态
- 模型版本
- 快照编号

### 筛选条件

- 日期区间
- 联赛
- 结算状态
- 模型版本
- 是否已锁定

## 7.5 统计分析页

### 目标

用基础统计证明平台是否具备长期参考价值。

### MVP 展示指标

- 总预测场次
- 已结算场次
- 胜平负命中率
- 让球胜平负命中率
- 近7天命中率
- 近30天命中率
- 近30天 ROI
- 累计 Yield
- 按联赛分布的样本数与命中率
- 按模型版本分布的样本数与命中率

### MVP 暂不纳入

- Log Loss
- Brier Score
- 概率校准图
- 赔率区间表现拆分
- 多模型对比图表

## 7.6 后台最小管理能力

### 目标

支撑 MVP 正常运行，不追求完整运营后台。

### 必要能力

- 同步体彩比赛池
- 同步赛果和状态
- 触发预测生成任务
- 查看任务执行日志
- 查看预测锁定状态
- 查看快照记录
- 查看结算结果

### MVP 不做

- 用户运营后台
- 公告 CMS
- 订单与支付后台
- 复杂权限系统

## 7.7 预测生成与发布

### 处理流程

1. 拉取当日体彩竞彩足球比赛池
2. 补充球队历史与基础特征数据
3. 调用模型生成胜平负与让球方向分析
4. 生成分析摘要与置信等级
5. 写入预测记录
6. 生成哈希签名
7. 发布到前台
8. 记录当日快照

### 输出内容

- 胜平负概率
- 让球胜平负倾向
- 预期总进球
- 模型置信等级
- 模型版本号
- 发布时间
- 哈希值

## 7.8 锁定与防篡改

### 核心规则

- 比赛开赛前可生成预测
- 到达锁定时间后，核心预测字段不可修改
- 已发布预测不可删除
- 如需补充复盘说明，只能追加备注

### MVP 必做能力

- 每条预测记录哈希值
- 每条预测记录发布时间
- 每日发布快照
- 审计日志记录关键操作

## 7.9 自动结算

### 目标

在比赛结束后，根据官方赛果自动完成结算。

### 支持市场

- 胜平负
- 让球胜平负

### 结算状态

- 命中
- 未中
- 走水
- 取消
- 延期

### 规则要求

- 结算逻辑由系统自动执行
- 不允许人工直接改写结算结果
- 异常场次只能通过补充状态解释，不覆盖原始记录

## 9. 技术建议

## 9.1 技术栈

- 前端: React + Vite
- 后端: Spring Boot
- 数据库: PostgreSQL
- 缓存: Redis
- 任务调度: Spring Schedule 或 Quartz
- 模型服务: Python 独立服务或离线批处理

## 9.2 MVP 架构建议

初期建议保持单体优先：

- 一个前端应用
- 一个后端应用
- 一个数据库
- 一个缓存实例
- 一个离线模型任务入口

说明:

- 模型部分可以先离线计算后入库，不必第一天就拆独立在线推理服务

## 10. 非功能要求

### 10.1 性能

- 比赛列表页响应时间目标小于 2 秒
- 比赛详情页响应时间目标小于 2 秒
- 历史记录筛选页响应时间目标小于 3 秒

### 10.2 可靠性

- 数据同步任务失败需可重试
- 预测生成任务必须有执行日志
- 快照生成失败需触发告警

### 10.3 安全性

- 后台接口必须鉴权
- 核心预测字段必须防止开赛后被修改
- 审计日志不可物理删除

## 11. 合规要求

### 11.1 产品边界文案

必须使用以下方向的表述：

- 数据分析
- 概率预测
- 风险提示
- 历史统计
- 模型结果仅供参考

必须避免以下表述：

- 稳赚
- 包赢
- 必中
- 跟单
- 内幕
- 无风险

### 11.2 页面必须具备

- 风险提示
- 免责声明
- 隐私说明

## 12. 验收标准

### 12.1 核心功能验收

- 能同步体彩当天竞彩足球比赛池
- 能生成当天比赛预测
- 能展示比赛列表和比赛详情
- 能展示发布时间、模型版本、哈希与快照编号
- 能在锁定后阻止修改核心预测字段
- 能自动结算胜平负与让球胜平负
- 能展示全量历史记录
- 能展示基础统计数据

### 12.2 信任机制验收

- 用户可看到每场预测的发布时间
- 用户可看到已结算和未命中的完整样本
- 用户可追溯到当日快照
- 后台保留关键操作审计日志

## 13. MVP 之后的扩展方向

- 用户注册与收藏
- 会员订阅
- 总进球、比分、半全场等玩法
- 概率校准与高级模型评估
- 赔率与盘口变化分析
- API 开放平台
