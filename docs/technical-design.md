# 竞彩罗盘技术设计草案

## 当前阶段

本文件用于承接 `requirements-mvp.md` 的实现设计，当前先明确设计范围和优先级，后续按模块逐步展开。

数据源选型、职责边界和验证标准见 `data-sources.md`。在体彩源和亚盘源通过两周样本验证前，不冻结业务表结构和同步接口。

## 优先设计主题

- 体彩比赛池同步流程
- 预测发布与锁定机制
- 自动结算规则
- 快照与审计日志设计
- 前后端接口边界

## 数据源架构原则

- 体彩 Provider 负责官方比赛池、比赛编号、官方让球、SP、销售状态和最终赛果
- Asian Odds Provider 负责亚洲让球盘口和主客赔率快照
- 两类 Provider 通过 `match_source_mappings` 显式关联，不共享或猜测业务主键
- 供应商原始响应先写入 `raw_data_payloads`，解析成功后再进入领域表
- 原始响应和盘口快照只追加，不覆盖历史记录
- Provider Adapter 负责外部字段转换，Controller 和公开接口不得直接返回供应商原始 JSON
- 同步任务必须记录调用额度、响应状态、解析状态、重试次数和数据延迟

### 建议同步链路

1. 同步体彩当日比赛池并建立内部 `match`
2. 根据当日联赛集合查询亚盘源，避免无关调用消耗额度
3. 标准化联赛、球队和开赛时间，生成候选映射
4. 高置信度映射自动确认，低置信度映射进入人工复核
5. 写入亚盘首次可见和封盘前快照
6. 比赛结束后以体彩或确认的权威赛果执行自动结算
7. 生成覆盖率、延迟、额度和异常统计
